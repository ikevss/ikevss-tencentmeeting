# 腾讯会议·邀约洞察 v2.0

> 基于 `@tencentcloud/tmeet` v1.0.12 · OAuth2 授权 · 个人和企业账号通用 · 会前→会中→会后 全生命周期覆盖。

---

## 目标用户

- 主要：需要频繁组织会议、追踪行动项的 **管理者 / IR 从业者 / 销售负责人**
- 次要：个人用户轻量建会
- 典型场景：融资路演后的投资者关系复盘、客户续约拜访复盘、周会/1on1 决策追踪

---

## 状态变量

多步操作中，AI 必须在上下文中保持以下变量，不可在步骤间丢失：

| 变量 | 来源 | 生命周期 |
|------|------|---------| 
| `meeting_id` | `tmeet meeting create` | 建会 → 邀请 → 复盘 |
| `meeting_code` | `tmeet meeting create` | 建会 → 邀请(ICS) |
| `join_url` | `tmeet meeting create` | 建会 → 邀请(邮件) |
| `record_file_id` | `tmeet record list` | 查录制 → 复盘 |
| `ctk` | agently-cli 第一阶段 | 5 分钟有效 |
| `ctk_ts` | `date +%s` 时间戳 | 判断 ctk 过期用 |
| `organizer_email` | `agently-cli +me` | 会话内持久 |

**上下文切换规则**：如果用户在中途切换话题（如从建会跳到查录制），AI 应保留已有变量但明确告知用户当前流程状态。

---

## 核心原则

1. **先检查再做事** — 每条请求进入操作前必须先通过前置检查（见下节 ⚡ 前置检查）
2. **创建完就问** — 成功创建会议后必须问"需要发邮件邀请吗？"
3. **发送前先确认** — 邮件必须浏览器预览 → 用户确认 → 才发送（agently-cli 两阶段确认）
4. **时间自动转** — 用户说"明天下午3点" → 自动转为 `2026-07-23T15:00+08:00`
5. **场景路由** — 用户的话决定进入哪个阶段，不是每次都全走：
   - 说"建/创建/预约/开会" → 阶段 1（建会），完成后提示进入阶段 2
   - 说"邀请/邮件/通知" → 阶段 2（生成邀请函 + 发送）
   - 说"录制/转写/纪要/复盘/回顾/分析" → 阶段 3（会后复盘）
   - 在前阶段完成后，主动问下一步
   - 复合意图（如"建会+通知张三"）按顺序执行阶段 1→2，不截断

---

## 多会议消歧义

当用户指示词（"明天/上次/那场/路演"）匹配到 ≥2 场会议时：
1. 列出候选（主题 · 时间 · 参会人数），让用户选编号
2. 用户确认后再继续操作
3. 不要自行假设"最近那场"

---

## 时间锚点

时间转换以**用户当前话语时刻**为基准，不以 skill 激活时刻为基准。

## 不做什么

- 不管理企业微信/飞书/Zoom/Teams 的会议
- 不在用户说"不用"之后继续推销邮件邀请
- 不跨过 agently-cli 两阶段确认私自发送
- 不修改 SuperIR 项目的任何文件
- 不在用户查录制时不问复盘直接结束
- 复盘时不生成除 HTML 报告之外的其他格式文件
- 复盘时不依赖钉钉AI听记、飞书妙记等外部听记服务

---

## ⚡ 前置检查

每次激活先跑：

```bash
tmeet auth status
```

未登录 → `tmeet auth login`（自动弹浏览器），等授权完成后继续。

## 对话操作流程

### [阶段 1] 创建会议
- **入口守卫**：通过前置检查（tmeet auth status 已登录）

- 0. tmeet 未装 → `npm install -g @tencentcloud/tmeet@v1.0.12`
- 1. 确认参数（主题/时间/时长） → `tmeet meeting create` → 记录 `meeting_code, join_url, meeting_id`
- 2. 🆕 主动问："需要给参会人发邮件邀请吗？精美响应式邀请函 + 日历附件"

### [阶段 2] 邮件邀请
- **入口守卫**：持有阶段1的 `meeting_id`/`meeting_code`/`join_url`；agently-cli 已安装已授权

**首次使用（检测到 agently-cli 未安装或未授权）：**

- 安装: `npm install -g @tencent-qqmail/agently-cli@2.1.0`
- 安装: `npm i react-email@3.0.7`
- 安装 skill: `npx skills add https://agent.qq.com --skill -g -y`
  （预期 skill 名称: agently-mail，安装后运行 `agently-cli +me` 验证）
- 授权: `agently-cli auth login`
- 确认: `agently-cli +me` 获取邮箱地址（记为 `organizer_email`）

**每次发送（已安装后的完整流程）：**

- 1. 询问收件人邮箱（如上下文中已有参会人邮箱则跳过，直接进入步骤 2）
- 2. 用 react-email 组件生成 HTML 邮件 → `docs/.tmp/邮件邀请-{meeting_id}.html`
  - 品牌色 `#0066FF`, max-width 600px, 全部 inline styles
- 3. `start ""` 打开浏览器预览 → "确认无误后我发送"
- 4. 手写 ICS (RFC 5545, CRLF 换行, UTC 时间) → `docs/.tmp/meeting-{meeting_id}.ics`
  - `ORGANIZER` 使用 `agently-cli +me` 返回的邮箱
- 5. agently-cli 两阶段发送 (`--body-file ... --attachment ...`)
  - 第一阶段 → 拿到 ctk → 记录 `ctk` 和 `date +%s` 时间戳 → 展示摘要 → **停下**
  - 用户确认后 → 第二阶段（需重新获取 ctk 如已过期）
  - 错误码按 agently-cli 错误码表处理

**半成品交付策略（邮件链路失败时）：**

如果邮件发送失败（安装失败/授权失败/发送超时），AI 必须输出：
> "✅ 会议已创建（会议号: xxx，入会链接: xxx）\n⚠️ 邮件邀请未发送（原因: xxx）\n手动通知：可将以上会议信息复制转发给参会人。"

### [阶段 3] 会后复盘
- **入口守卫**：无前置依赖，可独立触发

**前置判断 — AI 根据用户意图决定是否进入复盘：**

- 用户说"查录制/查转写/查纪要/智能纪要" → 拿到数据后主动问："需要对这场会做复盘分析吗？我可以生成可视化洞察报告。"
- 用户说"复盘/分析/回顾这场会/会开了什么" → 直接进入复盘流程

**复盘流程（三阶段引擎）：**

```
转写原文 ──▶ [Phase A 内容画像] ──▶ ⏸ 确认 ──▶ [Phase B 布局规划] ──▶ [Phase C 智能渲染] ──▶ HTML 报告
                │
                └── 并行拉取：transcript-get + smart-minutes + participants（同时执行，不串行等待）
```

**Phase A — 内容画像（信息完整度评估）**

| 等级 | 识别条件 | 视觉策略 |
|------|---------|---------|
| **高** | 含说话人标签+时间戳的逐字稿 | 全部视觉品质释放，信息图丰富 |
| **中** | 逻辑连贯的段落笔记，缺乏精确标签 | 标准视觉，推定项显式标注 |
| **低** | 三五句话、口头描述、回忆式重述 | **降级视觉精致度**：减信息图数量，增大标注字号和对比度 |

**关键规则：信息完整度越低，视觉越朴素。禁止用精美的信息图包装空洞的内容。**

**会议类型识别（动态路由）：**

| 特征关键词 | 识别为 | 分析视角 |
|-----------|--------|---------|
| "站会/sync/每日/morning" | Daily Standup | 精简（最多 4 章），无章节详解 |
| "周会/weekly/双周" | Weekly Sync | 标准 + 章节详解 |

| "战略/strategy/规划/半年/年度" | Strategic | 丰富 + SWOT + 象限 + 效率评分卡 |
| "客户/路演/sales/pitch" | Client Meeting | 商务浅色主题 + 满意度评估 |
| "跟进/follow-up/CRM/pipeline" | Follow-up | 行动导向：分布 + 时间线 + 客户状态 |

| "一对一/1on1/沟通/反馈/绩效/辅导" | Communication | 七类沟通盲点识别 + 改进追踪 |
| "脑暴/讨论/发散/创意/方案/探索" | Creative | 跨学科视角 + 意外洞见 + 故事关联 + 创作方向 |
| 不明显/混合 | General | 标准布局 |

**数据密度评估：**

| 指标 | 低密度 | 中密度 | 高密度 |
|------|--------|--------|--------|
| 行动项数量 | 0-3 | 4-10 | 10+ |
| 决策数量 | 0-2 | 3-6 | 7+ |
| 讨论主题数 | 1-2 | 3-6 | 7+ |
| 参会人数 | 1-2 | 3-5 | 6+ |
| 时长 | <15min | 15-45min | >45min |

**情感基调检测：**

| 基调 | 识别词 | 主题选择 |
|------|--------|---------|
| 积极 | 增长、突破、完成、上线 | light-green |
| 中性 | 讨论、了解、同步 | light-blue（默认） |
| 紧急 | 问题、阻塞、延迟、风险 | light-amber |
| 严肃 | 事故、合规、审查、损失 | light-slate |

**Phase A 输出示例（必须暂停等用户确认）：**

```
📊 内容画像
  完整度：中 | 类型：客户拜访 | 密度：中 | 基调：积极
  统计：5 行动项 · 3 决策 · 4 主题 · 3 参会者 · ~30min
  布局：标准模式 · 6 个章节 · light-blue 主题
  健康度评分：使用深度 ████░ 满意度 █████ 关系 ████░ 续约 █████ 增购 ███░░

是否继续生成完整报告？
```

**用户确认后再进入 Phase B。** 如果用户要求调整画像中的任何参数，据此修改后再继续。

**动态章节选择（根据数据量决定，不强行填充）：**

| 章节 | 触发条件 |
|------|---------|

| 对比分析 | 有明确对比内容（如与上期会议对比） |
| SWOT | 类型为战略/复盘/客户拜访 |
| 健康度评分卡 | 类型为客户拜访/客户成功 |
| 决策质量评分 | 类型为决策复盘 |
| 沟通盲点分析 | 类型为沟通复盘 |
| 跨学科洞见 | 类型为创意复盘 |
| 效率评分卡 | 完整度 ≥ 中 |
| 原文引用支撑 | 完整度 ≥ 中 |

**密度模式：**

| 模式 | 最大章节数 | 触发 |
|------|-----------|------|
| compact | 4（Hero + 2 核心 + 行动） | 低信息量 或 站会/1:1 |
| normal | 8 | 默认 |
| rich | 12 | 高信息量 或 战略/复盘/客户会议 |

**信息图节奏：连续信息图不超过 3 张。** 第 4 个起交替表格或结构化文本。

**Phase C — 智能渲染（纯 HTML/CSS 零外部 JS）**

**报告结构（根据会议类型动态组装）：**

```
[Hero 信息卡片] → [核心结论] → [关键决策] → [行动项] → [会议总览] → [时间线] → [场景洞察] → [原文引用]
```

**场景洞察区域（按会议类型自动适配）：**

| 会议类型 | 洞察维度 | 可视化组件 |
|---------|---------|-----------|
| 客户拜访 | 五维健康度（使用深度/满意度/关系稳定性/续约意愿/增购潜力）| 雷达图 + 趋势对比 + 风险信号列表 + 增购机会 |
| 决策复盘 | 决策类型（快速/深思/犹豫/推迟）+ 决策依据（数据/直觉/他人/风险）+ 质量评分 | 决策树 + 依据分布 + 盲区识别 + 改进建议 |
| 沟通复盘 | 七类盲点（表达模糊/打断/话题跳跃/过度解释/情绪化/被动回应/绝对化）| 频次分布 + 场景差异 + 改进目标 + 追踪对比 |
| 创意复盘 | 跨学科视角（心理学/哲学/社会学等）| 学科卡片 + 意外洞见 + 故事关联 + 创作方向 |
| 投资者关系 | Q&A 分析 + 技术展望 + 投资者反馈 | 问答分类 + 关注热度 + 后续跟进 |
| 通用会议 | 效率评分（议程达成率/决策产出率/参与均衡度/行动明确度）| 评分卡 + 改进建议 |

**诚实分层（必须严格遵守）：**

| 标注 | 含义 | 使用场景 |
|------|------|---------|
| `[原文引用]` | 直接引用，未做任何语言转换 | 逐字稿中完整、清晰的句子 |
| `[转写]` | 口语转为书面语，保留原意和完整事实 | 去除填充词、修正语法后的版本 |
| `[概括]` | 多句话合并为一条结构化信息 | 一段讨论浓缩为一句话结论 |
| `[推断]` | 从语境间接推导，原始话语中未明确表达 | 必须标注置信度 |

**规则：** 尽力使用 `[原文引用]` 和 `[转写]`，谨慎使用 `[概括]`，`[推断]` 必须标注置信度。不得将口语重构伪装为原文引用。缺失数据明确标记。

**品牌规范：** 品牌色 `#0066FF`，max-width 800px，table-based 布局，全部 inline CSS，无外部 JS 依赖。浅色背景 + 编辑级排版。

**CSS 禁止模式：** `border-left: Xpx solid COLOR` 搭配 `border-radius`（直角色条无法贴合圆角，产生视觉裂隙）。替代方案：`border-top: 3px solid COLOR`（顶边色条）、微妙 `box-shadow`、彩色编号文字或标签徽章。

**可访问性：** HTML 报告 `<html lang="zh-CN">`、图表给出文字摘要、颜色不作为唯一信息载体。

**会议冲突检测逻辑：**

```
用户说"帮我建一个明天下午3点的会"时：
1. 先查 `tmeet meeting list` 获取用户已有会议列表
2. 检测时间重叠：新会议时间区间与已有会议是否冲突
3. 如有冲突，主动提示：

   "⚠️ 明天下午3点您已有会议：
    - 14:00-15:00 周会（参会人：张三、李四）
    - 15:00-16:00 产品评审（参会人：王五、赵六）
    仍要创建吗？"

4. 用户确认后再执行创建
```

**并行执行策略：**

```
原流程（串行）：
  transcript-get → smart-minutes → participants（3 次串行 API 调用）

优化后（并行）：
  同时发起 transcript-get + smart-minutes + participants（1 轮并行 API 调用）
  
实现方式：
  AI 在单个回合内同时执行多条 Bash 命令（用 && 分隔或并行调用），
  收集全部结果后再进入 Phase A 内容画像。
```

**转写内容过长时：**

如果 `transcript-get` 返回超过 10000 字 → 告知用户：
> "这场会转写内容较长（xxx字），建议重点复盘某一个议题。你最关心这场会的哪个部分？"

**场景不明确时：**

如果无法从转写内容推断场景 → 问用户：
> "这场会是什么类型？① 投资者路演 ② 客户拜访 ③ 内部决策会 ④ 一对一沟通 ⑤ 脑暴讨论 ⑥ 通用复盘"

**用户拒绝下一步时：**

如果用户说"不用/不需要"，AI 应回复"好的，如果需要其他帮助随时说"并结束，不继续推销。

**复盘完成后的闭环：**

- 主动问："需要把复盘报告通过邮件发给参会人吗？" → 可回到阶段 2（发送）
- 如果是系列会议（如每周例会）：主动建议"需要和上次同类会议做趋势对比吗？"

## 关键约束

### 时间格式
ISO 8601 带时区：`2026-07-23T15:00+08:00`。默认时区 `+08:00`。
常见中文时间表达转换规则：
- "明天下午3点" → 当前日期+1天，15:00:00+08:00
- "今天下午3点" → 如果当前时间已过15:00则报错提示，否则用今天
- "下周一上午10点" → 计算下一个周一的日期
- "本月15号下午2点" → 当前年月 + 15日
- 用户未指定年份 → 默认当年

### HTML 邮件
全部 inline styles，Container max-width 600px，品牌色 `#0066FF`。底部署名区说明 .ics 附件双击导入日历。

### ICS 日历
```
BEGIN:VCALENDAR
VERSION:2.0
PRODID:-//ikevss-tencentmeeting//CN
METHOD:REQUEST
BEGIN:VEVENT
UID:{meeting_id}@tencentmeeting
DTSTART:{utc_start}
DTEND:{utc_end}
DTSTAMP:{now_utc}
SUMMARY:{subject}
DESCRIPTION:腾讯会议号：{meeting_code}\n入会链接：{join_url}
LOCATION:{join_url}
ORGANIZER;CN={organizer_name}:mailto:{organizer_email}
STATUS:CONFIRMED
END:VEVENT
END:VCALENDAR
```
行分隔 `\r\n`，时间转 UTC，`UID=meeting_id`，`{organizer_email}` 来自 `agently-cli +me`。

### agently-cli 两阶段确认（必须遵守）
1. 第一阶段拿 ctk → **记录 `ctk` 和当前时间戳**（`date +%s`） → 展示 summary → **停下等用户回复**
2. 绝不自己确认自己
3. ctk 5 分钟有效 → 重试或超时后**必须重新走第一阶段**（旧 ctk 不可复用）

### agently-cli 错误码

| exit | 含义 | 处理 |
|------|------|------|
| 0 | 成功 | — |
| 1 | 服务端/网络抖动 | 最多重试 2 次 |
| 2 | 参数不合规 | 不重试，按 error.message 改参数 |
| 3 | 授权失效 | 重新 `agently-cli auth login` |
| 4 | 本地网络 | 最多重试 2 次 |
| 6 | 业务拒收 | 不重试，反馈用户 |
| 7 | 限频 | 等 Retry-After 后重试 |
| 8 | 缺 confirmation-token | 走两阶段确认 |

## CLI 命令速查

| 场景 | 用户的话（示例） | 命令 |
|------|-----------------|------|
| 创建会议 | "帮我建一个明天下午3点的会" | `tmeet meeting create --subject ... --start ... --end ...` |
| 周期会议 | "每周一早上的周会" | 同上 + `--meeting-type 1 --recurring-type 2` |
| 查会议列表 | "我最近有哪些会" | `tmeet meeting list` |
| 取消会议 | "取消明天的会" | `tmeet meeting cancel --meeting-id ...` |
| 查录制/纪要 | "上次路演的录制和纪要" | `tmeet record list` → `tmeet record smart-minutes` |
| 查参会人 | "这场会谁来了" | `tmeet report participants --meeting-id ...` |
| 搜通讯录 | "搜通讯录里投资者关系的人" | `tmeet contact search --keyword ...` |
| 踢人 | "把不在名单的人踢出去" | `tmeet control kick --meeting-id ...` |
| 导出日志 | "导出最近7天的日志" | `tmeet tshoot log --start ... --end ...` |
| 复盘会议 | "帮我复盘上次路演" | `tmeet record list` → `tmeet record transcript-get` → 生成 HTML 报告 |

> 如需精细参数（`--invitees-type` `--page-token` `--auto-record-type` 等），运行 `tmeet <command> --help` 查看最新帮助。

## 降级处理

| 问题 | 解决 |
|------|------|
| tmeet 未安装 | `npm install -g @tencentcloud/tmeet@v1.0.12` |
| 未登录 | `tmeet auth login`，等浏览器授权 |
| 凭证过期 | 重新 `tmeet auth login` |
| 时间格式错误 | 必须 ISO 8601 带时区偏移 |
| SSH 无浏览器 | `tmeet auth login --no-browser`，手动打开链接 |
| agently-cli 发送失败 | 输出会议信息让用户手动通知（半成品交付） |
| npm install 卡住（>3 分钟） | 询问用户是否重试或跳过 |
| tmeet auth login 超时（>5 分钟） | 提示"授权未完成，可稍后重试"，进入降级 |
| agently-cli 发送超时（>2 分钟） | 按 exit code 1/4 处理（最多重试 2 次） |
| 录制未开启转写 | `tmeet record smart-minutes` 用智能纪要替代转写原文 |
| 转写内容过长 (>10K字) | 提示用户指定复盘议题，聚焦分析某一节 |
| 智能纪要也不可用 | 仅基于会议主题+参会人信息生成简单复盘卡片 |
| 录制需密码 | 用 `--pwd` 传录制文件密码 |
| 分页数据 | 用 `--page-token` 获取下一页 |
