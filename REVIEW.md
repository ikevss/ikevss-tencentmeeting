## 1. Skill 定位（8/10）

### ✅ 做得好
- 中文名「腾讯会议·邀约洞察」+ 英文名 `ikevss-tencentmeeting` 双命名，辨识度高。
- **"做什么"清晰**：建会 → 邀请函 → 邮件通知 → 复盘洞察，全生命周期覆盖明确。
- **"不做什么"明确**：在 description 和正文「不做什么」两处都列出了 NOT for 范围（企业微信/Zoom/Teams/飞书会议、实时转写、硬件管理）。
- **差异化价值主张突出**：在标题区显式点出"与腾讯官方 MCP 方案的核心区别：企业账号也能用"——这句话是卖点，放在最醒目位置，很好。

### ⚠️ 可改进
- **缺少目标用户画像**：当前只说"个人和企业账号通用"，但没有说清**为谁**做的。是投资人关系（IR）从业者？销售？HR？还是通用办公人群？从复盘场景（投资者关系/客户拜访/决策/1on1/脑暴）反推，似乎偏向**管理者和商务人群**，但 skill 没有明说。用户画像缺失会导致触发词覆盖和场景识别的设计缺乏锚点。
- **"不做什么"与 description 中的 NOT for 有轻微重复**：description 末尾已写 `NOT for 企业微信会议、Zoom、Teams、飞书会议；实时转写分析；会议室硬件管理。`，正文「不做什么」节又列了 7 条，部分重叠。建议 description 只保留 1 句精炼版，详细边界放正文。

### 📌 建议
```markdown
## 目标用户画像（建议新增）
- 主要：需要频繁组织会议、追踪行动项的 **管理者 / IR 从业者 / 销售负责人**
- 次要：个人用户轻量建会
- 典型场景：融资路演后的投资者关系复盘、客户续约拜访复盘、周会/1on1 决策追踪
```

## 2. Prompt 架构（8/10）

### ✅ 做得好
- **结构层次分明**：状态变量 → 核心原则 → 前置检查 → 对话操作流程（3 阶段）→ 关键约束 → CLI 速查 → 示例 → 降级 → 安全。阅读动线清晰。
- **版本锁定完整**：`@tencentcloud/tmeet@v1.0.12`、`@tencent-qqmail/agently-cli@2.1.0`、`react-email@3.0.7` 都锁定具体版本，避免漂移。
- **场景路由清晰**：核心原则第 5 条将用户话语映射到阶段 1/2/3，复合意图（"建会+通知张三"）明确按顺序执行不截断，很好。

### ⚠️ 可改进（冗余检测）
检测到 **2 处指令冗余**：

| 冗余点 | 位置 A | 位置 B | 建议 |
|--------|--------|--------|------|
| 前置检查重复声明 | 第 102-110 行「⚡ 前置检查」| 第 117-119 行阶段 1 步骤 0 又重复"通过前置检查" | 阶段 1 步骤 0 删除，仅保留入口守卫引用 |
| 场景路由重复 | 第 83-88 行核心原则第 5 条 | 第 157-160 行阶段 3 前置判断再次列关键词 | 保留核心原则 5 条做总览，阶段 3 只写"按核心原则第 5 条路由" |

### ⚠️ 可改进（指令歧义）
- **阶段 2 步骤 2 的"用 react-email 组件生成 HTML 邮件"**——没有指定**哪个** react-email 组件（Body? Button? Container?），也没有给出组件库的引用来源。AI 在执行时可能用错组件或写错 import。建议补一个最小组件清单或指向 react-email 官方文档。
- **阶段 2 步骤 1 "询问收件人邮箱"**——用户已在"建会+通知张三"复合意图里给了邮箱，这条规则会导致 AI 重复询问已明确的信息。应加一句"如上下文已有收件人则跳过"。

## 3. 工程质量（8/10）

### ✅ 做得好
- **错误码表完整**：agently-cli 的 0/1/2/3/4/6/7/8 错误码都有对应处理（重试/改参数/重新授权/等 Retry-After 等），非常实用。
- **状态变量表设计清晰**：`meeting_id`、`meeting_code`、`join_url`、`record_file_id`、`ctk`、`ctk_ts`、`organizer_email` 都标注了来源和生命周期，跨步骤传递不易丢。
- **降级链完整**：邮件失败 → 半成品交付（给用户手动转发信息）；转写不可用 → 用智能纪要；智能纪要也不可用 → 仅基于主题+参会人出简单卡片。**三层降级**，覆盖充分。
- **诚实分层**（阶段 3 步骤 6）：`[原文引用] > [转写] > [AI概括] > [AI推断]` 四级标注，**这是本 skill 最亮点的工程设计**，有效防止 AI 幻觉冒充事实。

### ⚠️ 可改进
- **缺少超时处理**：`npm install -g`、`tmeet auth login`（等浏览器授权）、`agently-cli` 两阶段发送都没有超时兜底。如果 npm 卡住或用户离开浏览器没授权，AI 会无限等。建议加"超过 3 分钟无响应 → 询问用户是否重试或降级"。
- **缺少并发冲突处理**：如果用户同时发"帮我复盘 A 会"和"帮我复盘 B 会"，两个 `record_file_id` 在上下文中可能互相覆盖。建议加"每场会议用 `meeting_id` 命名空间隔离变量"。
- **依赖文档链接缺失**：正文引用了 `tmeet`、`agently-cli`、`react-email` 三个关键依赖，但只有 tmeet 在末尾给了 GitHub 链接，**agently-cli 和 react-email 没有文档链接**。AI 遇到参数细节（如 `--invitees-type`、`--page-token`）只能靠 `--help`，但 react-email 的组件用法完全没有参考。
- **状态变量表中 `ctk` 生命周期描述"5 分钟有效"**：这其实是约束（constraint），不是生命周期。生命周期应该是"阶段 2 步骤 5 第一阶段 → 步骤 5 第二阶段前"。建议拆分清楚。

## 4. Frontmatter 质量（8/10）

### ✅ 做得好
- `name`、`version`、`description`、`author`、`url` 五要素齐全。
- `description` 同时覆盖"做什么"和"NOT for"，对自动路由友好。
- `triggers` 覆盖了约 **40 个触发词**，覆盖面非常广，从"建一个会议"到"会开了什么"都有，召回率应该不错。
- `version` 用了语义化版本 `"2.0.0"`。

### ⚠️ 可改进
- **author 字段缺少联系方式**：当前只写 `author: 花生ss`，建议加邮箱或 GitHub 用户名便于问题反馈，如 `author: 花生ss (github.com/ikevss)`。
- **description 略冗长**（约 80 字）：同时包含"做什么"、"NOT for"、"个人和企业账号通用"、"OAuth2 授权"。建议压缩到 1 句话 + NOT for，细节放正文。例如：
  ```yaml
  description: >
    腾讯会议·邀约洞察 — 覆盖建会、邀请函、邮件通知、会后复盘洞察全生命周期。
    NOT for 企业微信/Zoom/Teams/飞书会议、实时转写分析、会议室硬件管理。
  ```
- **triggers 中有轻微重复**："腾讯会议"和"tencent meeting"语义重叠；"会议纪要"、"腾讯会议纪要"、"智能纪要"三选二即可。精简后更利于精确匹配。

## 5. 多轮对话设计（8/10）

### ✅ 做得好
- **阶段衔接机制完整**：
  - 阶段 1 完成 → 主动问"需要发邮件邀请吗？"
  - 阶段 3 查录制后 → 主动问"需要复盘分析吗？"
  - 阶段 3 复盘完成 → 主动问"需要邮件发送给参会人吗？"
  - 形成**闭环**。
- **复合意图处理**：明确"建会+通知张三"按顺序执行阶段 1→2 不截断，避免 AI 只完成一半。
- **中途切换话题规则**：第 73 行"保留已有变量但明确告知用户当前流程状态"，这条很实用。
- **场景不明确时的消歧义**（第 183-186 行）：无法推断场景时主动列出 6 个选项让用户选，而不是 AI 瞎猜。

### ⚠️ 可改进
- **缺少多会议消歧义**：用户说"明天的会"但明天有 3 场会、或说"上次路演"但一个月路演了 5 次，AI 该选哪场？当前 skill 没给消歧义策略。建议加：
  > 当用户指示模糊（"那个会""上次""明天"）且匹配到多于 1 场会议时，列出候选会议（主题+时间+参会人数）让用户确认。
- **缺少时间锚点机制**：第 80 行说"用户说'明天下午3点'自动转"，但如果用户在长对话中间隔 3 小时再说"明天下午3点"，这个"明天"到底是哪个时间基准？建议加"以用户当前话语时刻为基准，不以 skill 激活时刻为基准"。
- **缺少"用户拒绝下一步"后的收尾**：第 93 行说"不在用户说'不用'之后继续推销邮件邀请"，但没说拒绝后 AI 该做什么——是直接结束？还是问"还有其他需要吗？"。建议补一句。

## 6. Top 5 改进建议

### 🔴 P0（必须修，影响正确性/一致性）

#### P0-1：阶段 1 步骤 0 与「前置检查」节重复，删除冗余
- **位置**：第 117-119 行
- **问题**：阶段 1 的"入口守卫"和"步骤 0"都在说"通过前置检查"，而第 102-110 行已经有完整的「⚡ 前置检查」节。三次说同一件事，浪费 token 且容易让 AI 困惑"到底该在哪一步检查"。
- **修改建议**：
  ```
  阶段 1 入口守卫：tmeet auth status 已登录（前置检查见第 102 行）
  步骤 0 删除，步骤 1 改为步骤 0，后续顺延
  ```

#### P0-2：阶段 2 步骤 1 与复合意图冲突，加"如上下文已有收件人则跳过"
- **位置**：第 138 行
- **问题**：用户说"建会+通知张三（zhangsan@x.com）"，按复合意图走阶段 1→2，但阶段 2 步骤 1 仍会"询问收件人邮箱"——用户已经给了，重复询问体验差。
- **修改建议**：
  ```
  1. 询问收件人邮箱（如上下文中已有参会人邮箱则跳过，直接进入步骤 2）
  ```

### 🟡 P1（强烈建议，影响鲁棒性）

#### P1-1：增加多会议消歧义策略
- **位置**：建议在「核心原则」或「阶段 3 前置判断」新增一条
- **问题**：用户说"复盘上次路演"但匹配到多场路演时，AI 可能随机选一场，结果复错了。
- **建议新增**：
  ```markdown
  ## 多会议消歧义
  当用户指示词（"明天/上次/那场/路演"）匹配到 ≥2 场会议时：
  1. 列出候选（主题 · 时间 · 参会人数），让用户选编号
  2. 用户确认后再继续操作
  3. 不要自行假设"最近那场"
  ```

#### P1-2：增加超时/卡死处理
- **位置**：建议在「降级处理」表或「关键约束」节新增
- **问题**：`npm install -g`、`tmeet auth login`（等浏览器授权）、agently-cli 发送都可能卡住。当前 skill 没有超时兜底，AI 会无限等待。
- **建议新增**：
  ```markdown
  ## 超时处理
  | 操作 | 超时阈值 | 超时后动作 |
  |------|---------|-----------|
  | npm install | 3 分钟 | 询问用户是否重试或跳过 |
  | tmeet auth login | 5 分钟 | 提示"授权未完成，可稍后重试"，进入降级 |
  | agently-cli 发送 | 2 分钟 | 按 exit code 1/4 处理（最多重试 2 次） |
  ```

#### P1-3：补充依赖文档链接
- **位置**：建议在文末「后续升级」前新增一节或补到现有链接
- **问题**：agently-cli 和 react-email 无文档链接，AI 遇到细节只能猜或 `--help`。
- **建议新增**：
  ```markdown
  ## 参考文档
  - [@tencentcloud/tmeet](https://github.com/TencentCloud/tencentmeeting-cli) v1.0.12
  - [@tencent-qqmail/agently-cli](https://github.com/TencentQQMail/agently-cli) v2.1.0
  - [react-email](https://react.email/) v3.0.7
  ```

### 🟢 P2（锦上添花，可后续迭代）

#### P2-1：复盘报告的可访问性（a11y）建议
- **位置**：阶段 3 步骤 6 的「品牌规范」旁
- **问题**：HTML 报告全部 inline CSS、无外部 JS，适合邮件，但缺少可访问性提示（如 `lang="zh-CN"`、图片 alt、语义标签）。
- **建议新增**：
  ```markdown
  - **可访问性**：HTML 报告 `<html lang="zh-CN">`、图表给出文字摘要、颜色不作为唯一信息载体
  ```

#### P2-2：复盘后的持久化/趋势追踪轻量指引
- **位置**：阶段 3 步骤 8 之后或「后续升级」节
- **问题**：「后续升级」提到了"复盘趋势追踪"，但当前版本没有指引 AI 如何把历次复盘关联起来。用户连续 5 周复盘周会时，AI 不会主动做对比。
- **建议新增**：
  ```markdown
  ## 复盘持久化（当前版本轻量实现）
  - 每次复盘 HTML 报告归档到 `docs/.tmp/复盘报告-{meeting_id}-{日期}.html`
  - 用户提到"和上周比趋势"时，主动读取上一份同主题报告做对比
  - 趋势追踪的完整版本将在后续升级实现
  ```

## 审查结论

**整体评价：生产可用（Production Ready）** ✅

本 skill 在"做什么/不做什么"、版本锁定、状态变量管理、错误码表、降级链、诚实分层（防幻觉）等方面做得相当扎实，尤其是**四级诚实分层**（原文引用 > 转写 > AI概括 > AI推断）和**三层降级链**（转写 → 智能纪要 → 简单卡片）是行业最佳实践级别的设计。

主要改进空间集中在：
1. **冗余清理**（P0，约 5 分钟可修）
2. **消歧义策略**（P1，多会议、多场路演场景必备）
3. **超时兜底**（P1，避免 AI 卡死）
4. **依赖文档链接**（P1，减少 AI 猜测）

修完 P0 + P1 后，本 skill 可达到 **9.0/10 的优秀水平**。
